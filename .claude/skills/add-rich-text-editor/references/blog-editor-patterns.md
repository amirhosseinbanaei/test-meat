# Blog editor patterns

Tips and patterns specific to building the blog-composition surface.

## Toolbar vs. bubble vs. slash menu — pick one (or compose them)

| Surface | When to use |
|---|---|
| **Fixed toolbar** (top of editor) | Default. Discoverable. The template uses this. |
| **Bubble menu** (appears near selection) | Pairs well with the toolbar for inline marks. Add `@tiptap/extension-bubble-menu` and conditionally render formatting buttons when text is selected. |
| **Slash menu** (`/heading`, `/image`, `/list`) | Modern blog editor UX. Best when the toolbar would crowd the chrome. Implement with `@tiptap/extension-mention` configured for `/` trigger, or build with `tippy.js`. |
| **Floating menu** (appears at start of empty line) | For block-level inserts when on a new line. `@tiptap/extension-floating-menu`. |

For a polished blog editor: fixed toolbar (always available, accessible) + bubble menu (fast inline formatting) + slash menu (block inserts on empty lines).

## Image embedding — wire to the upload skill

When the user clicks the image button or pastes/drops an image:

1. Show the file picker → run through `add-file-upload` (drop zone, compression, optional crop).
2. On upload success, get back a `publicUrl`.
3. Insert into the editor: `editor.chain().focus().setImage({ src: publicUrl, alt: '' }).run()`.

The template's `onRequestImage` prop is the hook for this — the parent component owns the upload flow and returns the URL. Keeps the editor itself free of upload concerns.

## Drag-and-paste images

Tiptap's Image extension doesn't intercept drag/paste of image files by default — they appear as base64-encoded `<img>` in the document, which is huge and wrong. Override:

```ts
Image.extend({
  addOptions() {
    return {
      ...this.parent?.(),
      inline: false,
    };
  },
  // Intercept paste/drop of image FILES — push them through the upload flow
  // instead of the default base64 embedding.
  addProseMirrorPlugins() {
    return [
      // Custom plugin that listens for paste/drop, finds image files in the
      // clipboard data, kicks off uploads, and inserts placeholder nodes
      // until the URL resolves. Implementation is non-trivial — easier to
      // just disable paste/drop of images and require the toolbar button.
    ];
  },
});
```

Pragmatic default: disable drag/paste of image files. Users add images via the toolbar button. Cleaner, easier to reason about.

## Auto-save

Long-form writing without auto-save is a UX trap. Pattern:

```tsx
const [draft, setDraft] = useState<Json | null>(initialDraft);
const debouncedSave = useDebouncedCallback(async (json: Json) => {
  await saveDraftAction(json);
}, 800);

<{{EDITOR_COMPONENT_NAME}}
  value={draft}
  onChange={(json) => {
    setDraft(json);
    debouncedSave(json);
  }}
/>
```

Show the save state in the toolbar — `Saving…`, `Saved`, `Save failed`. Users want to know.

Implement debouncing with a small custom hook or `useDebouncedCallback` from `use-debounce` (1 KB, no deps). Don't pull a full state library for this.

## Word / character count

Add `@tiptap/extension-character-count`. The editor exposes `editor.storage.characterCount.characters()` and `.words()`. Useful for limit enforcement and "X words read" labels.

## Headings → table of contents

For a blog post page, a TOC sidebar is generated by walking the stored JSON tree, filtering for heading nodes:

```ts
function tocFromJson(json: Json): Array<{ level: number; text: string; id: string }> {
  const toc: Array<{ level: number; text: string; id: string }> = [];
  function walk(node: any) {
    if (node?.type === 'heading') {
      const text = (node.content ?? []).map((n: any) => n.text ?? '').join('');
      toc.push({ level: node.attrs?.level ?? 1, text, id: slugify(text) });
    }
    (node?.content ?? []).forEach(walk);
  }
  walk(json);
  return toc;
}
```

Inject the same `id` attribute into the rendered HTML so anchor links work. Tiptap's StarterKit doesn't add IDs to headings by default — extend the Heading node with an `id` attribute that derives from the text content, or do the id injection in `generateHTML` post-processing.

## Accessibility

The editor area is `role="textbox"` (Tiptap sets this automatically). Beyond that:

- **Every toolbar button must have an `aria-label`.** Icons alone aren't accessible. The template does this.
- **Buttons that toggle state need `aria-pressed`.** The template does this.
- **The editor needs visible focus styles.** The `prose` class + the container's `focus-within:ring-2` will do it.
- **Keyboard shortcuts.** Tiptap's StarterKit binds the standards (`Cmd+B`, `Cmd+I`, etc.). If you add custom commands, document them in a help dialog.

## Mobile

Tiptap works on mobile, but the touch experience for selection + format is rough. For a blog editor specifically (desktop-first task), don't optimise for mobile. Show a "best edited on a larger screen" hint if the viewport is too narrow.

For a mobile-friendly editor — slash commands, large tap targets, single-purpose buttons — consider a stripped-down variant: no bubble menu, big-icon toolbar, sticky to the bottom of the viewport.

## What NOT to do

- **Don't store HTML.** JSON gives you safety, structure, and migration. HTML gives you sanitisation problems and lock-in to the current schema.
- **Don't use `dangerouslySetInnerHTML` with arbitrary user-pasted HTML.** The renderer in this skill is safe because `generateHTML` serialises through the schema. Random HTML from elsewhere is not safe.
- **Don't load the editor into the page header for SEO.** The renderer (Server Component) is for the public-facing display. Tiptap's editor goes only on the authoring routes.
- **Don't lazy-load the editor naively.** It pulls in a lot of code. Use `next/dynamic` with `ssr: false` for the editor component when it's not on the first paint.
